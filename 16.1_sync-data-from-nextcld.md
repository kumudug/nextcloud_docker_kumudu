# Sync data from Dockerized Nextcloud to the file server (rsnapshot)

This is the Docker-era replacement for [16_sync-data-from-nextcld.md](./16_sync-data-from-nextcld.md).
Full design rationale: [docs/superpowers/specs/2026-06-21-rsnapshot-nextcloud-docker-design.md](./docs/superpowers/specs/2026-06-21-rsnapshot-nextcloud-docker-design.md).

## What this does

* `rsnapshot` runs on the **gembi** file server and pulls each Nextcloud user's
  `files/` directory over SSH from the **Docker host**.
* Keeps **7 daily / 4 weekly / 6 monthly** hardlinked snapshots.
* Read access on the Docker host is granted by adding `syncusr` to the **gid-33**
  group that owns the container-written data — **no `chown`/`sgid` on the data**.

## Parameters

| Thing | Value |
|-------|-------|
| Docker host | `192.168.4.2`, SSH port `28`, admin login `kumudu` |
| Data path (Docker host) | `/data/nextcloud/nextcloud_data/<user>/files/` |
| Nextcloud users | `kumudu`, `kumudu_amazon` |
| Snapshot root (gembi) | `/databackup/nextcld_docker/` |
| rsnapshot config (gembi) | `/etc/rsnapshot-nextcld.conf` (separate from the existing `/etc/rsnapshot.conf`) |

## Step 1 — Create the sync key on gembi

* Create the account that holds the key
   - `sudo useradd -g users -m -s /bin/bash syncusr`
* Generate a dedicated keypair (kept as `id_rsa` to match the config below)
   - `sudo -u syncusr ssh-keygen -t ed25519 -N "" -f /home/syncusr/.ssh/id_rsa -C "syncusr@gembi rsnapshot nextcloud"`

  > **Why this command, flag by flag:**
  > - **Dedicated keypair** — a key made *solely* for this backup job, not reused
  >   from any personal/admin login. If it leaks or you need to revoke backup
  >   access, you delete just this one key without disturbing anything else.
  > - **Kept as `id_rsa`** — the file on disk is named `id_rsa` because Step 4's
  >   verify commands and the `ssh_args` line in Step 5 hardcode that exact path
  >   (`-i /home/syncusr/.ssh/id_rsa`). Rename it and those `-i` paths break.
  > - **Note the naming quirk:** `-t ed25519` makes this an **Ed25519** key, *not*
  >   an RSA key. `id_rsa` is just the conventional "default private key"
  >   filename, reused here so the config paths stay simple. Modern key type,
  >   old-convention filename.
  > - `sudo -u syncusr` — runs as `syncusr` so the key files are owned by that
  >   account from the start (no later `chown` needed).
  > - `-N ""` — empty passphrase, required because the backup runs unattended from
  >   a systemd timer (Step 7); a passphrase would need someone to type it.
  > - `-f /home/syncusr/.ssh/id_rsa` — output path; creates `id_rsa` (private) and
  >   `id_rsa.pub` (public).
  > - `-C "syncusr@gembi rsnapshot nextcloud"` — a comment label embedded in the
  >   public key so you can identify it later in `authorized_keys`.
* Print the public key — **copy this line**, you need it in Step 3
   - `sudo cat /home/syncusr/.ssh/id_rsa.pub`

## Step 2 — Create the gid-33 group and syncusr on the Docker host

* Check whether gid 33 is already taken
   - `getent group 33`

  > **On this host, gid 33 already exists as `http`** (Arch's web group; it's the
  > host-side name for the container's `www-data`/uid 33), so `getent group 33`
  > prints `http:x:33:` and the steps below use `http`. If instead nothing is
  > printed, create the group first (`sudo groupadd -g 33 http`); if some *other*
  > name is already at gid 33, use that name in place of `http` below. The
  > group name doesn't need to match gembi — Step 5's `--groupmap=*:users`
  > remaps group ownership during the pull.

* Create the sync user in that group
   - `sudo useradd -g users -G http -m -s /bin/bash syncusr`
   - `sudo passwd syncusr`
* Verify membership (look for `33(http)` in the output)
   - `id syncusr`

## Step 3 — Trust the key and allow SSH on the Docker host

* Create the `.ssh` directory
   - `sudo install -d -m 700 -o syncusr -g users /home/syncusr/.ssh`
* Add the public key from Step 1 into `authorized_keys`
   - `sudo vim /home/syncusr/.ssh/authorized_keys`  (paste the `ssh-ed25519 …` line)
   - `sudo chown syncusr:users /home/syncusr/.ssh/authorized_keys`
   - `sudo chmod 600 /home/syncusr/.ssh/authorized_keys`
* Allow `syncusr` over SSH — pick **one** of the two approaches below
   - See what you have first: `sudo sshd -T | grep -iE '^(allowusers|allowgroups)'`

  **Approach A — `AllowGroups` (recommended):** grant access via group
  membership. sshd matches on a user's primary *or* supplementary group.
   - ⚠️ **Do not add `syncusr` to an admin group.** On this host `AllowGroups`
     is `wheel` (the sudo group); adding `syncusr` there would give the
     read-only backup account full sudo. Instead, add a **dedicated, non-privileged
     group** and extend the line:
   - `sudo groupadd sshsync`
   - `sudo usermod -aG sshsync syncusr`
   - `sudo vim /etc/ssh/sshd_config`  →  extend the line to `AllowGroups wheel sshsync` (keep `wheel` so existing admins aren't locked out)
   - Make sure there's no `AllowUsers` line — if both `AllowUsers` and
     `AllowGroups` are present, a user must satisfy **both**.

  **Approach B — `AllowUsers` (explicit per-user list):**
   - `sudo vim /etc/ssh/sshd_config`  →  set `AllowUsers kumudu syncusr` (remove any `AllowGroups` line)

  Then, either way:
   - `sudo sshd -t && sudo systemctl restart sshd`

  > Keep `PasswordAuthentication no`. `syncusr` logs in with the Ed25519 key from
  > Step 1; the `passwd syncusr` set in Step 2 only affects local `sudo`/console,
  > not SSH.

* Confirm the directive
   - `sudo sshd -T | grep -iE '^(allowusers|allowgroups)'`
   - (Approach A) verify the membership took: `id syncusr`

## Step 4 — Verify read access (do not skip — this is the gate)

Run these **from gembi**:

* Confirm `rsync` is installed on the Docker host — **required on both ends**;
  rsnapshot runs a remote copy of rsync over SSH, and if it's missing the transfer
  dies with `rsync returned 12` (nothing to do with keys or config)
   - `sudo ssh -i /home/syncusr/.ssh/id_rsa -p 28 syncusr@192.168.4.2 'which rsync'`
   - Expect a path like `/usr/bin/rsync`. If it prints nothing, install it on the
     Docker host: `ssh -p 28 kumudu@192.168.4.2 'sudo pacman -S --needed rsync'`
* List each user's files (should succeed, no password prompt)
   - `sudo ssh -i /home/syncusr/.ssh/id_rsa -p 28 syncusr@192.168.4.2 'ls /data/nextcloud/nextcloud_data/kumudu/files/'`
   - `sudo ssh -i /home/syncusr/.ssh/id_rsa -p 28 syncusr@192.168.4.2 'ls /data/nextcloud/nextcloud_data/kumudu_amazon/files/'`
* Confirm writes are denied (backup account must be read-only) — expect `READONLY_OK`
   - `sudo ssh -i /home/syncusr/.ssh/id_rsa -p 28 syncusr@192.168.4.2 'touch /data/nextcloud/nextcloud_data/kumudu/files/__write_test 2>&1 || echo READONLY_OK'`

> If `ls` gives `Permission denied`: re-check group membership and that the parent
> dirs (`/data`, `/data/nextcloud`, …) are traversable. A fresh SSH session is
> needed for new group membership to take effect.

## Step 5 — Install and configure rsnapshot on gembi

> **This host already runs rsnapshot** for local `nas/` backups. Do **not** edit
> the existing `/etc/rsnapshot.conf` — `snapshot_root`, `retain`, `rsync_long_args`
> and `lockfile` are all global, so mixing the Nextcloud job into it would collide
> with (and could corrupt the ownership of) the existing backups. Instead use a
> **dedicated config file** with its own snapshot root and lockfile, invoked with
> `-c` on every command.

* Install (if not already) and create the snapshot root
   - `sudo pacman -S --needed rsnapshot`
   - `sudo install -d -m 700 /databackup/nextcld_docker`
* Create a **new** dedicated config (leave the existing `/etc/rsnapshot.conf` alone)
   - `sudo vim /etc/rsnapshot-nextcld.conf`
* Put these values in it. **Every gap between fields is a literal TAB, never spaces:**

```
config_version	1.2
snapshot_root	/databackup/nextcld_docker/
cmd_cp	/usr/bin/cp
cmd_rm	/usr/bin/rm
cmd_rsync	/usr/bin/rsync
cmd_ssh	/usr/bin/ssh
cmd_logger	/usr/bin/logger

# must differ from the other config's lockfile or the two jobs block each other
lockfile	/var/run/rsnapshot-nextcld.pid
logfile	/databackup/logs/rsnapshot-nextcld.log

retain	daily	7
retain	weekly	4
retain	monthly	6

ssh_args	-i /home/syncusr/.ssh/id_rsa -p 28
rsync_long_args	--delete --numeric-ids --relative --usermap=*:syncusr --groupmap=*:users

backup	syncusr@192.168.4.2:/data/nextcloud/nextcloud_data/kumudu/files/	nextcld/kumudu/
backup	syncusr@192.168.4.2:/data/nextcloud/nextcloud_data/kumudu_amazon/files/	nextcld/kumudu_amazon/
```

* Validate the new config (expect `Syntax OK`)
   - `sudo rsnapshot -c /etc/rsnapshot-nextcld.conf configtest`

## Step 6 — Take the first snapshot and verify

* Dry run (prints the commands it would run)
   - `sudo rsnapshot -c /etc/rsnapshot-nextcld.conf -t daily`
* Real run
   - `sudo rsnapshot -c /etc/rsnapshot-nextcld.conf daily`
* Verify the snapshot has content
   - `sudo ls /databackup/nextcld_docker/daily.0/nextcld/kumudu/ /databackup/nextcld_docker/daily.0/nextcld/kumudu_amazon/`
   - `sudo du -sh /databackup/nextcld_docker/daily.0/`

## Step 7 — systemd service + timer units on gembi

Create three services and three timers. Higher intervals fire **earlier** so
rsnapshot rotation promotes a snapshot before the lower interval rotates.

> **Unit names are prefixed `rsnapshot-nextcld-*`** to avoid clashing with any
> existing rsnapshot cron/systemd jobs for the local `nas/` backups.

* `/etc/systemd/system/rsnapshot-nextcld-daily.service`

```ini
[Unit]
Description=rsnapshot daily snapshot of Nextcloud data

[Service]
Type=oneshot
ExecStart=/usr/bin/rsnapshot -c /etc/rsnapshot-nextcld.conf daily
```

* `/etc/systemd/system/rsnapshot-nextcld-weekly.service` — same, but `ExecStart=/usr/bin/rsnapshot -c /etc/rsnapshot-nextcld.conf weekly`
* `/etc/systemd/system/rsnapshot-nextcld-monthly.service` — same, but `ExecStart=/usr/bin/rsnapshot -c /etc/rsnapshot-nextcld.conf monthly`

* `/etc/systemd/system/rsnapshot-nextcld-daily.timer`

```ini
[Unit]
Description=Run rsnapshot daily (Nextcloud)

[Timer]
OnCalendar=*-*-* 03:00
Persistent=true

[Install]
WantedBy=timers.target
```

* `/etc/systemd/system/rsnapshot-nextcld-weekly.timer` — same, but `OnCalendar=Mon *-*-* 02:30`
* `/etc/systemd/system/rsnapshot-nextcld-monthly.timer` — same, but `OnCalendar=*-*-01 02:00`

* Reload systemd
   - `sudo systemctl daemon-reload`

## Step 8 — Enable timers and verify

* Enable and start all three
   - `sudo systemctl enable --now rsnapshot-nextcld-daily.timer rsnapshot-nextcld-weekly.timer rsnapshot-nextcld-monthly.timer`
* Confirm they are scheduled (check the `NEXT` column)
   - `systemctl list-timers 'rsnapshot-nextcld-*' --all`
* Smoke-test one service through systemd
   - `sudo systemctl start rsnapshot-nextcld-daily.service`
   - `journalctl -u rsnapshot-nextcld-daily.service -n 20 --no-pager`

## Restore

Snapshots live at `/databackup/nextcld_docker/{daily,weekly,monthly}.N/nextcld/<user>/`
(`.0` is newest). To restore:

* Copy files back out of a snapshot
   - `sudo rsync -a /databackup/nextcld_docker/daily.0/nextcld/kumudu/ /path/to/restore-staging/kumudu/`
* After placing files back into the live Nextcloud data dir (as the data owner /
  via the container), re-index them
   - `docker compose exec --user www-data app php occ files:scan --all`

## Troubleshooting

### `Host key verification failed` after rebuilding the Docker host

If the Docker host was reinstalled/rebuilt but kept the same IP, its SSH host key
changes while the old key is still cached, and you'll see:

```
Host key for [192.168.4.2]:28 has changed and you have requested strict checking.
Host key verification failed.
```

Because the backup runs as **root** (`sudo ssh`, and rsnapshot's `cmd_ssh` under
the systemd timer), the stale entry lives in **root's** `known_hosts`, not your
user's. Fix it there:

* Remove the stale entry (bracket + port form, because port is non-standard)
   - `sudo ssh-keygen -R "[192.168.4.2]:28"`
* Verify the new key is really the Docker host — on the **Docker host**, print its fingerprint
   - `sudo ssh-keygen -lf /etc/ssh/ssh_host_ed25519_key.pub`
* Reconnect from gembi and compare the fingerprint before accepting
   - `sudo ssh -i /home/syncusr/.ssh/id_rsa -p 28 syncusr@192.168.4.2 'id'`
   - If it matches, type `yes`; the new key is written to `/root/.ssh/known_hosts`.
* Or, to pre-seed non-interactively (only if you trust the network right now)
   - `sudo ssh-keyscan -p 28 -H 192.168.4.2 | sudo tee -a /root/.ssh/known_hosts`

> If you ran the failing `ssh` **without** `sudo`, the stale entry is in your own
> `~/.ssh/known_hosts` instead — drop the `sudo` from `ssh-keygen -R`.
